# 看板数据处理与页面展示全链路说明

本文档描述当前仓库中“数据读取 -> 指标计算 -> payload 生成 -> 页面展示”的真实实现路径，供后续排查和改动时快速对齐口径。

## 1. 总体架构

- 原始数据目录：`data/<门店>/`
- 后端核心：
  - 文件解析与标准化：`ingestion/file_resolver.py`、`ingestion/pipeline.py`
  - 周指标引擎：`core/metrics_engine.py`
  - 页面主 payload 生成：`core/dashboard_builder.py`
- 前端展示：
  - 页面入口：`src/App.tsx`
  - 只读 `data/warehouse/ui_payload.json`（前端不做业务计算）

## 2. 原始数据读取与文件解析

### 2.1 门店文件发现

`ingestion/file_resolver.py::resolve_store_files()` 会在门店目录下选“最新修改时间”的 xlsx 文件：

- 订单：`店内订单明细...全部订单` 或 `订单明细`
- 支付：`支付明细`
- 销售：`菜品销售明细` 或 `品项销售明细`（两种版本都支持）
- 评价：`评价管理` / `店内评价`
- 报损、团购、菜品库、品类映射

### 2.2 读取后标准化

`ingestion/pipeline.py` 的 `load_store_bundle()` 将单店读成 `StoreBundle`：

- `orders`：订单明细，补 `business_date`、`week_id`、`order_revenue`
- `payments`：补 `week_id`、付款人 `_payer`
- `sales_sold` / `sales_return`：
  - `sales_sold` 来自 `已销售` 或 `品项销售明细`
  - 若有 `退菜` sheet 则读入 `sales_return`
  - 收入列统一归一到 `菜品收入（元）`
- `reviews`：店内评价明细，支持多 header 行、`星级分` 等列
- `store_rating_sheet`：门店评分 sheet，支持：
  - 日期为 Excel 序列号
  - `实际值` 为文本（如“星级分 4.4，服务 4.5”）

## 3. 周指标计算（weekly_metrics）

`core/metrics_engine.py::compute_fresh_weekly_table()` 以订单周数据为主轴计算周指标：

- 核心：营收、订单、客单价、翻台率
- 团购、折扣、复购
- 门店评分（实际值）口径：
  1. 当周有门店评分 -> 取当周最后一条 `实际值`
  2. 当周无 -> 取最近一天 `实际值`
  3. 仍无 -> 回退评价明细周均分

`MetricsEngine` 会把 fresh 与 `data/warehouse/weekly_metrics.json` 历史合并（同店同周保留最新），形成趋势底表。

## 4. 页面 payload 生成（ui_payload）

`core/dashboard_builder.py::build_ui_payload()` 逐店逐周生成页面结构：

- `coreMetrics`
- `categoryAnalysis`
- `productDetails`（top/bottom、退换菜、报损）
- `timeAnalysis`
- `marketing`
- `service`（差评条数、评分、关键词、证据）
- `externalAndWeather`
- `summary`

### 4.1 历史冻结与重算边界

`write_ui_payload(..., recompute_from_week_id=...)` 控制周是否重算：

- 默认（不传）：已有周冻结，仅新增周计算
- 传 `recompute_from_week_id`：`week_id >= 边界` 强制重算，`< 边界` 继续冻结

### 4.2 差评统一口径（当前）

差评统计与关键词口径统一：

- 差评判定：`score <= 3`
- 差评关键词池：`score <= 3`
- 兜底：若差评文本存在但 NLP 没抽到关键词，使用文本首句生成关键词标签，避免“条数有但关键词空”

### 4.3 退/换菜统计口径（当前）

`_returns_count_week()` 支持两种销售导出：

1. 有独立 `退菜` sheet：按该表统计
2. 无 `退菜` sheet（新版品项销售明细）：从 `sales_sold` 识别退/换菜行：
   - `退菜数量 > 0`
   - 或 `退菜金额(元) > 0`
   - 或 `销售数量 < 0`
   - 或 `敏感操作类型` 包含“退菜|换菜”
   - 计数优先按 `订单号` 去重

## 5. 前端数据获取与展示

`src/App.tsx` 的关键逻辑：

- `fetchJsonFromMirrors()` 依次尝试：
  1. GitHub Pages
  2. jsDelivr
  3. raw.githubusercontent
- 用 `cache: "no-store"` 规避缓存旧包/旧 JSON
- `buildDashboardData()` 负责选门店、选周、拼页面结构

页面“六、服务与质量”使用字段：

- 差评条数：`service.negativeReviews.thisWeek`
- 门店评分：`service.rating.thisWeek`
- 差评关键词：`service.badKeywords`

## 6. 常见错位与排查顺序

1. **页面显示旧值**  
   先检查线上 `ui_payload.json` 与本地是否一致（不是先看代码）

2. **历史周不变**  
   是否被冻结；重算时需设置正确 `recompute_from_week_id`

3. **差评条数与关键词冲突**  
   先看 `service.negativeReviews` 与 `service.badKeywords` 是否同周同源；再看是否为冻结历史数据

4. **退/换菜全 0**  
   检查销售文件是否无 `退菜` sheet；若是新版“品项销售明细”，应走 `sales_sold` 回退逻辑
